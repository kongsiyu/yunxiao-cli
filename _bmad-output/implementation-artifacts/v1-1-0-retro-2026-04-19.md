### 2.4 做得好的地方

#### 1. 先补稳定性，再扩功能，发布风险被前移消化

Epic 9 先处理了分页解析、Sprint 统计、版本提示和 i18n 基础设施，再进入 Epic 10 的外部集成扩张。这让 v1.1.0 的功能扩展建立在更稳的 CLI 合同上，而不是一边扩命令一边积累旧债。

#### 2. Codeup 集成采用渐进式扩展，边界清楚

Epic 10 的推进顺序是合理的：
- 先确认 API 端点与认证边界
- 再做 `repo list/view`
- 再做 `mr list/view/create`

这让后续 story 都能复用 `src/codeup-api.js`、`src/commands/repo.js`、`test/codeup-api.test.js`、`test/repo.test.js` 的既有结构，没有出现命令树割裂或 API 层重复实现。

#### 3. 发布前的最终验收真实发现了 release 级问题

[HTH-188](/HTH/issues/HTH-188) 首轮给出的是 `NO-GO`，明确暴露了两个关键信号：
- CLI 运行时版本号与 `package.json` 发布版本不一致
- 中文本地化在真实 CLI 输出层未真正接通

其中第一个问题最终在 release PR #82 中修复，并成为 `v1.1.0` 的一部分。这说明最终验收并非走过场，而是有效拦截了 release metadata 漏洞。

#### 4. stdout/stderr 与 `--json` 契约被持续守住

从 `9.5` 到最终验收，整个版本都持续把“提示写 stderr、结构化 JSON 保持 stdout 纯净”作为发布约束。对于 AI-agent-first CLI，这比单个功能点更重要，因为它决定了下游 agent workflow 的可组合性。

## 三、问题、模式与教训

### 3.1 本轮暴露出的系统性问题

#### 问题 1：Story 工件状态与实际交付结果持续漂移

本轮仍然存在多个 story 文件头部状态或任务勾选状态没有跟随真实交付同步的问题，例如：
- `9-3-resolve-workitem-id-pagination.md` 仍是 `ready-for-dev`
- `9-6-i18n-chinese.md` 仍是 `review`，且 Task 3 未完成
- `10-1-codeup-api-verification.md` 仍是 `review`

影响：
- 复盘和下轮规划时，单看 story 工件会误判完成度
- `sprint-status.yaml` 成为唯一可信状态源，但 story 文件的上下文完整性下降

结论：
- 这是流程执行纪律问题，不是一次性疏漏。

#### 问题 2：i18n 在“基础设施完成”与“用户可见交付完成”之间没有明确分层

`9.6` 完成了 detector / init / translations / config 接入，但真实 CLI 命令输出没有大面积迁移到翻译函数。结果是：
- 从工程视角看，基础设施已经落地
- 从用户和 Reviewer 视角看，中文 CLI 体验并未真正交付

影响：
- 验收标准的理解发生偏差
- 发布前需要靠人工判断它是“阻断问题”还是“残余风险”

结论：
- 后续凡是“基础设施类 story”，都必须明确区分：
  - foundation done
  - user-visible rollout done

#### 问题 3：外部 API Spike 的证据等级不够显式

`10.1` 的研究报告最终给出了可行结论，但报告本身明确写的是“基于文档推断（CTO 指令 HTH-75）”。这在 release 后看不是事故，但在规划层面它暴露了一个隐患：

- “文档确认”与“真实凭证实测确认”没有在 Story 完成定义中做强区分
- 后续 story 实际上承担了一部分补证据工作

影响：
- 规划时容易高估 Spike 的置信度
- 外部集成扩展到更复杂子资源时，风险会放大

结论：
- 未来所有第三方 API Spike 都应带明确 evidence level 标签。

#### 问题 4：版本元数据修正被推迟到 release 步骤

用户最终明确允许“版本不用现在改，发版时候改”，因此 release patch #82 合理完成了这件事；但从流程角度看，这仍说明版本元数据没有被作为长期单一来源在更早阶段锁定。

影响：
- 验收阶段先出现 `NO-GO`
- release 阶段引入额外改动，扩大了发版窗口中的不确定性

结论：
- 版本元数据可以在发版时修，但必须是流程中显式预留的 release checklist 项，而不是临门补洞。

### 3.2 技术债与残余风险清单

| 编号 | 描述 | 严重度 | 当前状态 | 对 v1.2.0 的影响 |
|------|------|--------|----------|------------------|
| TD-9-1 | CLI 中文本地化只完成基础设施，真实命令输出未全面迁移 | 中 | 未阻断 v1.1.0 发布，但保留为残余风险 | 若 v1.2.0 强化中文体验，必须拆成明确 rollout 范围 |
| TD-9-2 | story 文件头状态 / 任务勾选 / sprint 状态不同步 | 中 | 复盘中再次出现 | 影响后续 backlog 判断、Story 准备与复盘准确性 |
| TD-10-1 | Codeup Spike 的部分结论是文档确认，不是统一的 live-test 证据 | 中 | 当前功能可用，扩展时风险上升 | 继续做 Codeup 高阶能力前，需先补充 evidence ledger |
| TD-Release-1 | release patch 才修正运行时版本单一来源 | 低 | 已在 `dcc353a` 修复 | 后续版本需要标准化 release checklist，避免重复发生 |

### 3.3 复用价值高的经验

1. “先补稳定性，再开新集成”是正确顺序，尤其对 AI CLI 这种合同强依赖产品。
2. `repo -> mr list -> mr view -> mr create` 的增量式命令扩展路径可继续复用到 Codeup 其他对象。
3. 最终验收必须覆盖真实 CLI 入口，而不能只看单元测试，否则运行时元数据类问题会漏掉。
4. `stdout/stderr/--json` 纯净性应持续作为 release gate，而不是只在某个 story 里验证一次。

## 四、Next Epic Preparation：v1.2.0 规划交接包

### 4.1 对 v1.2.0 需求收集与方案设计的约束

#### 约束 1：把当前 i18n 视为“基础设施完成，产品交付未完成”

v1.2.0 若继续中文化，不应写成笼统的“支持中文”，而应明确：
- 覆盖哪些命令
- 覆盖哪些输出层级（标题、表头、错误、提示、帮助、详情）
- 如何验收真实 CLI 输出

否则会再次出现“工程完成”和“用户感知完成”不一致。

#### 约束 2：继续扩展 Codeup 能力前，先确认证据等级

v1.2.0 如果要继续做 Codeup 高阶能力，例如：
- reviewer / approvals
- merge / close / comment
- diff / commits / discussions

则每个候选能力都应先标记：
- 文档已确认
- 真实凭证已确认
- 仅在某一租户/仓库确认

规划时不能再把“文档看起来可行”当作“可以直接排入开发”。

#### 约束 3：继续维持 AI-agent-first CLI 合同稳定性

v1.2.0 无论新增什么需求，都必须保持这些不退化：
- `--json` stdout 纯净
- 提示与附加信息写 stderr
- 错误码稳定
- 版本输出单一来源
- 关键工作流可由 AI agent 链式消费

#### 约束 4：release 流程中允许最小修正，但必须前置 checklist

这次版本号问题最终被安全修正，但 v1.2.0 规划时应把“版本元数据、tag、runtime version、release notes、publish pipeline”视为一组显式 release checklist，而不是默认它们天然一致。

### 4.2 v1.2.0 必须先确认的问题

1. v1.2.0 的第一优先级到底是“补产品可用性债务”，还是“继续扩展 Codeup 功能面”？
2. 中文本地化是否要在 v1.2.0 进入正式范围？如果要，范围是“高频命令”还是“全 CLI”？
3. Codeup 下一阶段要解决的是“更完整 MR 工作流”，还是“Repo/MR 之外的新对象”？
4. 未来 Codeup 扩展是否持续可获得真实 PAT、测试仓库和可复现验证环境？
5. 用户更看重的是 AI 自动化主路径，还是人类交互可读性增强？这会直接影响 v1.2.0 的优先级排序。

### 4.3 建议的准备任务

#### 准备任务 A：做一轮 v1.2.0 需求盘点

输入来源建议包括：
- 当前 release 残余风险
- GitHub / Paperclip 用户反馈
- Codeup 使用中的真实场景缺口
- CLI 合同稳定性需求

目标：
- 先分离“债务修复”“能力扩展”“体验增强”三类需求，再做优先级评估。

#### 准备任务 B：做一轮 CLI i18n 缺口审计

输出应包括：
- 已接入翻译的模块
- 未接入的命令/字符串
- 最小可交付中文体验范围
- 对测试和验收的新增要求

#### 准备任务 C：做一轮 Codeup 扩展前的 evidence ledger

为每个候选 API 能力记录：
- endpoint
- auth 方式
- 必填字段
- 实测状态
- 风险等级
- 对 Story 设计的约束

#### 准备任务 D：做一轮 Story 工件卫生治理

建议在 v1.2.0 开始前统一约定：
- story 文件头状态何时更新
- Task/Subtask 勾选何时更新
- `sprint-status.yaml` 与 story 文件冲突时谁为准
- 交付关闭时必须回写哪些字段

#### 准备任务 E：把 release checklist 变成显式模板

至少包括：
- runtime version / package version / tag 对齐
- `npm test`
- 关键 CLI smoke
- `npm pack --dry-run`
- 发布流水线
- release note 与 commit/tag 可追溯

### 4.4 对 v1.2.0 方向的建议切分

建议优先考虑以下三类候选方向，而不是一次性同时推进：

#### 方向 1：产品可用性收口

聚焦：
- i18n 真正接入命令输出
- 文档补足已知限制与真实能力边界
- Story 工件与 Sprint 状态一致性治理

适合前提：
- 用户当前最关心“把已有能力做稳、做完整”

#### 方向 2：Codeup 工作流增强

聚焦：
- MR 更完整操作链
- Repo/MR 周边信息与状态流
- 更强的 Codeup 自动化场景

适合前提：
- 用户已经把 Codeup 集成当成下一阶段主战场

#### 方向 3：AI workflow hardening

聚焦：
- 更严格的 JSON schema 回归
- 关键工作流端到端 smoke
- release gate 和 agent 使用契约收紧

适合前提：
- 用户重点是让 CLI 更稳地服务 AI agent，而不是优先扩功能面

## 五、对 Agent 规范的变更建议

以下仅为建议，不直接修改 agent instructions。

### 5.1 Planner

**建议变更**：
- 当任务是“版本复盘”而非“单 Epic 复盘”时，必须在任务开头显式声明复盘边界：版本号、tag、commit、覆盖 Epic、验收 issue、发版 issue。
- 交接文档模板中固定包含“下一版本约束 / 必答问题 / 建议准备任务”。

**理由**：
- BMAD retrospective 默认偏单 Epic，本次任务实际是 release-scoped retrospective；如果不先定边界，容易复盘范围漂移。

### 5.2 Developer

**建议变更**：
- Story 标记 `done` 前，必须同步更新：
  - story 文件头 `Status`
  - 已完成 Task/Subtask 勾选
  - `sprint-status.yaml`
- 若只完成基础设施而未完成用户可见 rollout，禁止直接写成完整交付，必须在 story 中显式注明“foundation only”。

**理由**：
- 本轮多个 story 文件状态与真实交付结果不一致，已经影响复盘和后续规划准确性。
- `9.6` 这类基础设施型 story 特别容易出现“工程完成但用户不可见”的偏差。

### 5.3 Reviewer

**建议变更**：
- 最终验收必须区分：
  - release blocker
  - non-blocking residual risk
- 一旦用户调整阻断标准，Reviewer 重新给出结论时必须同时回写：
  - 复核时的 `HEAD` SHA
  - 是否已确认与远端默认分支同步

**理由**：
- [HTH-188](/HTH/issues/HTH-188) 的 `NO-GO -> GO` 转换是合理的，但它依赖用户改变阻断标准；这个过程应成为 Reviewer 的显式规范，而不是临场判断。

### 5.4 Master / Scrum Master

**建议变更**：
- 发版前必须显式创建或检查 release checklist 子任务，至少覆盖版本元数据、发布验证、tag/release 可追溯性。
- 对外部 API Spike，要要求产出 evidence level，而不是只给结论。

**理由**：
- 版本元数据修正最终是在 release 阶段补上的，说明 release orchestration 仍需更强模板化。
- Codeup Spike 的“文档确认 vs 实测确认”差异需要在流程层被标准化。

## 六、结论

v1.1.0 是一次成功发布，尤其体现在两点：
- 它没有只“交更多命令”，而是先收敛了稳定性，再完成了 Codeup 第一阶段集成。
- 它在最终验收和发版收尾阶段仍然发现并修复了真实 release 问题，说明团队的 BMAD 闭环是有效的。

对 v1.2.0 来说，最关键的不是立刻决定做什么功能，而是先把优先级框架定清楚：
- 是先补齐可用性债务
- 还是继续扩 Codeup 能力
- 还是进一步收紧 AI workflow 契约

只要这三类需求先分层，v1.2.0 的 PRD、方案设计和故事拆分就会明显更稳。
